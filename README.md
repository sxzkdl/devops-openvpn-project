# devops-openvpn-project
# Финальная работа: Развертывание и автоматизация безопасной VPN-инфраструктуры

## Общее описание проекта
Данный проект выполнен в рамках финальной работы курса «Старт в DevOps». Цель работы - спроектировать и автоматизировать развертывание безопасной сетевой инфраструктуры для IT-компании. 

В ходе проекта была решена задача организации безопасного удаленного доступа сотрудников во внутреннюю сеть через VPN с использованием собственного Удостоверяющего Центра (PKI) для проверки SSL-сертификатов. Все этапы настройки серверов автоматизированы с помощью Bash-скриптов для обеспечения воспроизводимости инфраструктуры.

---

## Общие требования и инструменты
* **Облачная платформа:** Google Cloud Platform (GCP).
* **Базовая ОС:** Виртуальные машины Compute Engine на базе **Debian 11 (Bullseye)**.
* **Используемое ПО:** `easy-rsa`, `openvpn`, `iptables`.
* **Сетевая защита:** Настройка правил брандмауэра GCP (входящий порт **UDP 1194** для VPN) и локальных фаерволов на серверах.

---

## Архитектура и состав серверов

Проект реализован на базе двух изолированных виртуальных машин в облаке:

| Имя сервера | Роль в сети | Установленные компоненты |
| :--- | :--- | :--- |
| **ca-server** | Изолированный Удостоверяющий Центр (CA) | `easy-rsa` (инициализация PKI, выпуск корневого сертификата) |
| **vpn-server** | Шлюз OpenVPN | `openvpn` (маршрутизация трафика клиентов, NAT/Маскарадинг) |

## Выполненные задания и автоматизация

### 1) Задание 1: Развертывание Удостоверяющего Центра (PKI)
На первом этапе была развернута изолированная машина `ca-server`. Процесс инициализации инфраструктуры публичных ключей полностью автоматизирован.

* **Скрипт автоматизации:** `auto-ca.sh` - автоматически обновляет систему, устанавливает `easy-rsa`, конфигурирует файл переменных `vars` и выпускает корневой сертификат `ca.crt` в режиме `nopass`.
* **Артефакты (в папке screenshots/):**
  * `auto-ca.sh` - исходный код bash-скрипта.
  * `auto-ca_running.png` - скриншот вывода скрипта ~/auto-ca.sh.
  * `ca.cert_present_ls.png` - скриншот подтверждения успешной генерации корневого сертификата ca.crt.

### 2) Задание 2: Создание и настройка VPN-сервера
На втором этапе в той же сети развернута машина `vpn-server`. На ней настроен OpenVPN, принимающий шифрованный трафик от удаленных сотрудников.

* **Скрипт автоматизации:** `setup-vpn.sh` - автоматизирует установку OpenVPN, настраивает сетевой форвардинг пакетов (`sysctl`), конфигурирует правила фаервола и включает NAT (маскарадинг) через `iptables` для выпуска клиентов в интернет.
* **Облачная безопасность:** В панели брандмауэра GCP создано правило `openvpn-rule`, открывающее порт **UDP 1194** наружу для входящих подключений клиентов.
* **Артефакты (в папке screenshots/):**
  * `setup-vpn.sh` - исходный код скрипта настройки VPN.
  * `VM_instances.png` - скриншот проверки статусов виртуальных машин, а также внутренних и внешних IP-адресов.
  * `openvpn_rule.png` - скриншот настроенного правила брандмауэра `openvpn-rule` в консоли Google Cloud.
  * `openvpn_connected.png` - скриншот успешного подключения клиента в программе OpenVPN Connect.
  * `yandex_ip_check.png` - скриншот Яндекс.Интернетометра, подтверждающий изменение внешнего IP-адреса клиента на IP-адрес VPN-сервера.

### 3) Задание 3: Настройка мониторинга
На третьем эатпе создадим и настроим свою инфраструктуру для мониторинга с использованием Prometheus. Для начала спроектируем систему мониторинга:

а) Какие инфраструктурные метрики можно собирать с ОС? (Для всех ВМ)
Чтобы вовремя заметить, что сервер «умирает», мы будем собирать стандартные системные метрики с помощью утилиты Node Exporter (база для Prometheus):
   * 1) CPU Usage (Загрузка процессора): Метрика `node_cpu_seconds_total`. Позволяет увидеть, если какой-то процесс завис и потратил все ресурсы.
   * 2) Memory Usage (Оперативная память): Метрики `node_memory_MemAvailable_bytes` и `node_memory_MemTotal_bytes`. Нужны, чтобы сервер не упал от нехватки памяти (OOM Killer).
   * 3) Disk Space (Свободное место): Метрика `node_filesystem_free_bytes`. Если место на диске кончится, логи перестанут писаться, и OpenVPN или сам сервер просто перестанут запускаться.
   * 4) Network Traffic (Сетевой трафик): Метрики `node_network_receive_bytes_total` и `node_network_transmit_bytes_total`. Помогают отслеживать аномальные всплески трафика (например, DDoS-атаку).

b) Какие решения существуют для мониторинга OpenVPN?
Для съема метрик с VPN-сервера мы выбираем специальный экспортёр — OpenVPN Exporter (например, от kumina или flant). Он подключается к Status-логу OpenVPN или к его Management-интерфейсу и отдает в Prometheus следующие важные данные:
- Количество активных подключенных клиентов в данный момент (openvpn_active_clients).
- Объём трафика, проходящий через VPN-туннель (сколько байт получено/отправлено клиентами).
- Статус работы самого демона OpenVPN (работает или упал).

c) Какие алерты заводить для мониторинга ситуации с виртуальными машинами?
Пишем правила на языке PromQL, которые будут триггерить Alertmanager при опасности:
1) Сервер недоступен (Instance Down): `up == 0` (если Prometheus не может достучаться до Node Exporter больше 5 минут — паника, сервер выключился).
2) Кончается место на диске (Disk Space Low): `node_filesystem_free_bytes{mountpoint="/"}/node_filesystem_size_bytes{mountpoint="/" * 100 < 10` (свободно меньше 10% диска).
3) Процессор перегружен (High CPU Load): Загрузка CPU > 90% на протяжении 10 минут.

d) Какие алерты заводить, чтобы убедиться, что OpenVPN в порядке?
1) OpenVPN Service Down (Сервис упал): Если специальный флаг от OpenVPN Exporter равен нулю на протяжении 2-3 минут.
2) Аномальное количество клиентов (Too Many Connections): Если количество подключений резко превысило норму (сигнал о возможной компрометации ключей или брутфорсе).
