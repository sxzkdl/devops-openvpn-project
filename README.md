# devops-openvpn-project
# Финальная работа: Развертывание и автоматизация безопасной VPN-инфраструктуры

## Общее описание проекта
Данный проект выполнен в рамках финальной работы курса «Старт в DevOps». Цель работы - спроектировать и автоматизировать развертывание безопасной сетевой инфраструктуры для IT-компании. 

В ходе проекта была решена задача организации безопасного удаленного доступа сотрудников во внутреннюю сеть через VPN с использованием собственного Удостоверяющего Центра (PKI) для проверки SSL-сертификатов. Все этапы настройки серверов автоматизированы с помощью Bash-скриптов для обеспечения воспроизводимости инфраструктуры.

---

## Общие требования и инструменты
* **Облачная платформа:** Google Cloud Platform (GCP).
* **Базовая ОС:** Виртуальные машины Compute Engine на базе **Debian 11 (Bullseye)**.
* **Используемое ПО:** `easy-rsa`, `openvpn`, `iptables`.
* **Сетевая защита:** Настройка правил брандмауэра GCP (входящий порт **UDP 1194** для VPN) и локальных фаерволов на серверах.

---

## Архитектура и состав серверов

Проект реализован на базе двух изолированных виртуальных машин в облаке:

| Имя сервера | Роль в сети | Установленные компоненты |
| :--- | :--- | :--- |
| **ca-server** | Изолированный Удостоверяющий Центр (CA) | `easy-rsa` (инициализация PKI, выпуск корневого сертификата) |
| **vpn-server** | Шлюз OpenVPN | `openvpn` (маршрутизация трафика клиентов, NAT/Маскарадинг) |

## 3) Задание 3: Настройка и автоматизация мониторинга

### а) Проектирование системы и собираемые инфраструктурные метрики
Для обеспечения непрерывного контроля работоспособности виртуальных машин и своевременного обнаружения деградации производительности (состояния, когда сервер «умирает»), на всех целевых узлах развернут и изолирован `Node Exporter`. Архитектура мониторинга собирает следующие базовые системные метрики:

* **CPU Usage (Загрузка процессора):** * *Метрика:* `node_cpu_seconds_total`
    * *Назначение:* Позволяет осуществлять непрерывный мониторинг утилизации процессорных мощностей, выявлять аномальные долгоживущие процессы, утилизирующие 100% ядра, и предотвращать критическое замедление бизнес-логики.
* **Memory Usage (Оперативная память):** * *Метрики:* `node_memory_MemAvailable_bytes` и `node_memory_MemTotal_bytes`
    * *Назначение:* Необходимы для контроля дефицита свободной RAM и прогнозирования срабатывания механизма ядра Linux OOM Killer (Out-Of-Memory), который может принудительно завершить критические процессы системных служб.
* **Disk Space (Свободное место):** * *Метрика:* `node_filesystem_free_bytes{mountpoint="/"}`
    * *Назначение:* Контроль заполнения дисковой подсистемы. Переполнение хранилища приводит к невозможности ротации и записи системных логов, аварийной остановке СУБД, демона OpenVPN и самого Prometheus.
* **Network Traffic (Сетевой трафик):** * *Метрики:* `node_network_receive_bytes_total` и `node_network_transmit_bytes_total`
    * *Назначение:* Помогают отслеживать пропускную способность сетевых интерфейсов, фиксировать аномальные всплески трафика и своевременно реагировать на сетевые инциденты (например, DDoS-атаки).

### b) Архитектурное решение по мониторингу OpenVPN
В ходе реализации пусконаладочных работ инфраструктуры было принято решение отказаться от интеграции кастомных сторонних экспортеров (таких как экспортеры от *kumina* или *flant*). 

**Причина:** Использование непроверенных бинарных файлов из сторонних репозиториев без долгосрочной LTS-поддержки нарушает требования к безопасности и стабильности контура мониторинга. 

**Альтернативное решение:** Жизнедеятельность и статус работы самого демона OpenVPN, а также общая сетевая связность и доступность хоста `vpn-server` переведены на мониторинг с использованием метрики Node Exporter и встроенной функции Prometheus `up`. Сбор метрик настроен по внутреннему защищенному каналу. Если служба или сама VM с VPN падает, триггер алерта `Instance Down` срабатывает автоматически.

### c) и d) Настройка правил алертинга (PromQL)
Для автоматической фиксации внештатных ситуаций и оповещения администраторов в Alertmanager были интегрированы правила, написанные на языке PromQL (конфигурационный файл `/etc/prometheus/alert.rules.yml`):

1.  **Сервер недоступен (Instance Down)**
    * *Формула:* `up == 0`
    * *Критерий:* Prometheus не может достучаться до Node Exporter на любой из целевых ВМ на протяжении **1 минуты**.
    * *Логика:* Критический алерт (Severity: critical). Означает падение операционной системы, сбой по питанию или полную изоляцию сетевого интерфейса.
2.  **Кончается место на диске (Disk Space Low)**
    * *Формула:* `(node_filesystem_free_bytes{mountpoint="/"}/node_filesystem_size_bytes{mountpoint="/"}) * 100 < 10`
    * *Критерий:* Объем свободного пространства в корневом разделе хоста составляет **менее 10%** от общего размера диска в течение **5 минут**.
    * *Логика:* Предупреждающий алерт (Severity: warning). Сигнализирует о необходимости немедленной очистки логов или расширения диска до отказа служб.

### Автоматизация развертывания и структура скриптов
Вся процедура развертывания подсистемы мониторинга была полностью автоматизирована с помощью Bash-скриптов, расположенных в корне данного репозитория. Скрипты написаны с соблюдением стандартов POSIX и Debian Policy Manual (использованы флаги `set -e` для перехвата и остановки выполнения при любой ошибке кода).

1.  **`install-monitoring.sh`** - главный скрипт автоматизации для **сервера мониторинга**. 
    * *Что делает:* Создает изолированных системных пользователей (`prometheus`, `alertmanager`) без домашней директории и шелла (`/bin/false`). Создает структуру каталогов в `/etc` и `/var/lib`, скачивает официальные LTS-релизы Prometheus и Alertmanager, генерирует конфигурационные файлы, настраивает SMTP-шлюз для отправки уведомлений на Email, создает `systemd` юниты, выставляет корректные права (`chown`) и запускает службы в автозагрузку.
2.  **`install-node-exporter.sh`** - скрипт автоматизации для **целевых серверов (VM)**.
    * *Что делает:* Быстро разворачивает Node Exporter в качестве демона `systemd` на отслеживаемых виртуальных машинах для съема метрик.

## 4) Задание 4: Создание системы резервного копирования

### План реализации и архитектура системы бэкапирования
Для обеспечения высокой доступности (High Availability) и гарантированного восстановления данных в случае аварии, система резервного копирования проектируется по следующей схеме:

* **Выделенное хранилище (Backup Server):** В инфраструктуру добавляется новый изолированный сервер `backup-server-01` в альтернативной географической зоне (например, если основные сервера развернуты в `europe-west3-c`, то сервер бэкапов разворачивается в `europe-west1-b`). Это защищает данные от глобального сбоя конкретного дата-центра Google Cloud.
* **Локальный агент сборщика:** На целевых серверах (`vpn-server` и `monitoring-server`) разворачивается bash-скрипт автоматизации, упакованный в `.deb`-пакет. Скрипт выполняется по расписанию через `systemd-timer`.
* **Безопасная транспортировка:** Передача резервных копий на `backup-server-01` осуществляется по зашифрованному протоколу SSH с использованием утилиты `rsync` и авторизацией по SSH-ключам с ограниченными правами, исключая сквозную компрометацию сети.

### Анализ возможностей платформы (GCP) VS Cобственная реализация
Для достижения максимального уровня надежности применяется гибридный подход:

| Компонент / Задача | Платформа (GCP) | Собственная реализация | Обоснование |
| :--- | :--- | :--- | :--- |
| **Снапшоты дисков (Compute Engine Snapshots)** | Используется (раз в сутки) | — | Позволяет мгновенно пересоздать и запустить упавшую виртуальную машину целиком из готового образа. |
| **Бэкап Удостоверяющего Центра (CA) и ключей** | — | Скрипт + шифрование `gpg` | Облачные снапшоты не гарантируют консистентность файлов в момент их записи. Ключи VPN требуют изоляции и шифрования на уровне ОС. |
| **Хранение deb-пакетов и скриптов** | — | Hosted Git + локальный deb-репозиторий | Исключает привязку кода к локальному ноутбуку инженера. Обеспечивает воспроизводимость версий пакетов. |

### Документ анализа рисков (Вероятные сценарии отказа)

> **Принцип проектирования:** *Design for failure* — любая система рано или поздно откажет.
#### Ниже приведены 5 критических сценариев и регламент действий по их ликвидации.

#### Сценарий 1: Аппаратный сбой или удаление виртуалки `vpn-server`
* **Последствия:** Полная остановка процессов удаленного доступа, сотрудники теряют связь с закрытым контуром.
* **Действия по восстановлению:** Развертывание новой ВМ из последнего облачного снапшота (GCP Snapshot) в панели управления. В случае его недоступности - поднятие чистой ОС Ubuntu через мастер-скрипт `setup-vpn.sh` из Git-репозитория и накатывание бэкапа конфигурации CA с сервера бэкапов.

#### Сценарий 2: Компрометация или уничтожение приватного ключа Удостоверяющего Центра (CA)
* **Последствия:** Злоумышленник может выпустить валидные сертификаты для несанкционированного доступа, либо легитимные пользователи теряют доступ без возможности отзыва старых сертификатов.
* **Действия по восстановлению:** Немедленный отзыв скомпрометированного корневого сертификата. Развертывание нового CA из скрипта `auto-ca.sh`. Перевыпуск конфигураций (`.ovpn`) для всей базы пользователей.
* **Защита:** Хранение приватного ключа CA на сервере бэкапов в зашифрованном симметричным алгоритмом AES-256 виде.

#### Сценарий 3: Переполнение дискового пространства на сервере бэкапов (`backup-server-01`)
* **Последствия:** Новые точки восстановления не могут быть записаны, система мониторинга фиксирует деградацию подсистемы хранения.
* **Действия по восстановлению:** Срабатывает настроенный алерт `BackupDiskSpaceLow` в Prometheus. Администратор выполняет аудит ротации архивов. При необходимости увеличивается размер диска в панели GCP без остановки ВМ.

#### Сценарий 4: Сбой сети между зонами облака
* **Последствия:** Скрипты резервного копирования завершаются с ошибкой соединения, Prometheus генерирует алерт о недоступности таргета бэкапов.
* **Действия по восстановлению:** Скрипт автоматизации фиксирует ненулевой код возврата (`exit code != 0`), сохраняет бэкап в локальную временную директорию `/var/tmp/backup/` и отправляет повторный запрос при восстановлении связности.

#### Сценарий 5: Поломка рабочего ноутбука инженера-разработчика
* **Последствия:** Риск потери актуальных версий автоматизации, bash-скриптов развертывания и конфигурационных файлов.
* **Действия по восстановлению:** Инфраструктура защищена, так как все наработки (включая исходный код `.deb` пакетов) синхронизированы с удаленным Git-репозиторием. Развертывание рабочего окружения на новом устройстве сводится к выполнению команды `git clone`.

---
### Стратегия хранения скриптов и deb-пакетов
Для обеспечения надежного жизненного цикла инфраструктурного кода внедряются два контура хранения:

1. **Git-репозиторий (Контур исходного кода):** Все конфигурационные файлы и исходный код bash-скриптов хранятся в удаленном Git-репозитории. Это гарантирует версионирование, аудит изменений и защиту от потери локальных данных.
2. **Deb-репозиторий (Контур дистрибуции):** Собранные `.deb`-пакеты не хранятся россыпью. На выделенной машине разворачивается локальный apt-репозиторий. Это позволяет серверам обновлять утилиты мониторинга и бэкапов стандартным системным вызовом `apt-get update && apt-get upgrade`.
